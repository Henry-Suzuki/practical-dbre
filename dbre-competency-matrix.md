# DBRE (Database Reliability Engineering) Competency Matrix

単なる「技術力」だけでなく、「組織への影響力」や「デリバリー（自動化含む）」を重視しています。

### レベル定義の目安

* **E1 (Associate):** 指示を受けてタスクを遂行できる・学習段階
* **E2 (Intermediate):** 自律的に運用・開発ができる
* **E3 (Senior):** チームやプロジェクトをリードし、複雑な課題を解決できる
* **E4 (Staff):** 組織全体や複数チームにまたがる技術戦略を策定・推進できる

## 1. Technical Know-How (DB内部構造・アーキテクチャ・設計)

DBREとしての「守り」と「攻め」の技術力の深さです。

| Level | 期待される行動・スキル |
| --- | --- |
| **E1** | ・RDBMSの基本的なアーキテクチャ（プロセス、メモリ、ディスク）を理解している。<br>・既存のスキーマに対して、効率的なSQLクエリを作成できる。<br>・レビュアーの指導のもと、小規模なDDL変更を安全に実施できる。 |
| **E2** | ・インデックスの仕組み（B-Tree等）を理解し、EXPLAINを用いてクエリチューニングができる。<br>・レプリケーションの仕組みを理解し、遅延トラブル等の一次対応ができる。<br>・レプリケーションの仕組みを理解し、遅延トラブル等の一次対応ができる。<br>・正規化・非正規化のトレードオフを考慮した論理スキーマ設計ができる。 |
| **E3** | ・ロック、トランザクション分離レベル、MVCCの挙動を深く理解し、並行処理時のトラブルを解決できる。<br>・大規模トラフィックを想定したシャーディング、パーティショニング戦略を設計できる。<br>・NoSQLやNewSQLなど、ユースケースに応じた適切なデータストア選定ができる。 |
| **E4** | ・カーネルパラメータやファイルシステムレベルでのDBパフォーマンスチューニングができる。<br>・組織全体のデータアーキテクチャ（DWH/データレイク含む）の整合性とロードマップを策定できる。<br>・特定のDB製品に依存しない、抽象化されたデータアクセス層の設計・指針を示せる。 |

## 2. Reliability & Operations (信頼性・運用・SLO)

SREの文脈における「信頼性」への貢献度です。

| Level | 期待される行動・スキル |
| --- | --- |
| **E1** | ・アラート発生時に手順書（Runbook）に従って一次対応ができる。<br>・バックアップの取得状況を確認し、レポートできる。<br>・基本的なメトリクス（CPU, IOPS, Connection）の意味を理解している。 |
| **E2** | ・定型的なトイル（Toil）を認識し、スクリプト等で削減できる。<br>・過去の事象に基づき、監視アラートの閾値を適切に調整できる。<br>・インシデント発生時、ログやメトリクスから根本原因の特定（RCA）に参加できる。 |
| **E3** | ・SLO/SLIを定義し、エラーバジェットに基づいたリリース判断や信頼性向上の提案ができる。<br>・カオスエンジニアリングや防災訓練（DiRT）を計画・主導できる。<br>・複雑な障害において、他チーム（アプリ、インフラ）を巻き込んでインシデントコマンダーを務められる。 |
| **E4** | ・組織全体のアベイラビリティ戦略（マルチリージョン、DRサイト運用）を策定できる。<br>・「信頼性」をプロダクトの価値として定義し、経営層やPMに対してエンジニアリングリソースの投資対効果を説明できる。 |

## 3. Automation & Platform (自動化・IaC・開発者体験)

DBRE特有の「DBをサービスとして提供する（DBaaS）」能力です。

| Level | 期待される行動・スキル |
| --- | --- |
| **E1** | ・TerraformやAnsibleなどのIaCツールの基本的な構文を理解し、修正ができる。<br>・既存のCI/CDパイプラインを実行し、デプロイフローに従える。 |
| **E2** | ・DBのプロビジョニングや設定変更をIaCコードで記述し、レビューに出せる。<br>・マイグレーションツール（Flyway, sqldef等）を用いて、スキーマ変更の自動化を実装できる。<br>・手動オペレーションを廃止し、自動化されたワークフローに置き換えることができる。 |
| **E3** | ・開発者がセルフサービスでDBを利用できるプラットフォーム（社内DBaaS）を設計・構築できる。<br>・「Guardrails（ガードレール）」の概念を導入し、開発者が危険なクエリや設定を投入できない仕組みを作れる。<br>・プロビジョニング時間を短縮するための高度な自動化アーキテクチャを実装できる。 |
| **E4** | ・全社的なInfrastructure as Codeの設計思想やポリシー（モジュール構成、State管理）を標準化できる。<br>・開発ベロシティとデータベースの安全性を両立させるための、組織レベルでのツールチェーン選定と導入ができる。 |

## 4. Communication & Culture (文化・教育・影響力)

DBREとしての「啓蒙活動」です。Henryさんが重視されている部分かと思います。

| Level | 期待される行動・スキル |
| --- | --- |
| **E1** | ・自分が行った作業や学びをドキュメント化し、チーム内で共有できる。<br>・わからないことを適切に質問し、チームの知見を活用できる。 |
| **E2** | ・アプリ開発者からのDB相談（クエリレビュー依頼等）に対して、根拠を持ってアドバイスができる。<br>・ポストモーテム（事後振り返り）に参加し、再発防止策を文書化できる。 |
| **E3** | ・「DBRE」の概念をチーム外に広め、アプリ開発チームが自律的にDBパフォーマンスを意識できる文化を作れる。<br>・若手メンバーのメンターとなり、技術的な成長を支援できる。<br>・社内外での登壇やテックブログ執筆を通じて、組織の技術プレゼンスを向上できる。 |
| **E4** | ・組織間のサイロを破壊し、開発・インフラ・QAが一体となってデータ信頼性を担保する体制を構築できる。<br>・採用基準の策定や技術ブランディングを主導し、優秀なエンジニアを惹きつけることができる。<br>・業界のトレンドを把握し、自社に導入すべき技術と捨てるべき技術の判断ができる。 |
